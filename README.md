# Язык программирования (Forth)

## 1. Синтаксис

Грамматика языка задаётся следующей EBNF-нотацией.

```ebnf
letter      = "A" | "B" | ... | "Z" | "a" | "b" | ... | "z" ;
digit       = "0" | "1" | ... | "9" ;
identifier  = letter, { letter | digit } ;
number      = digit, { digit } ;
string      = '"', { character - '"' }, '"' ;
comment     = "(", { comment | character - "(" - ")" }, ")"
            | "\", { character - newline }, newline ;
whitespace  = space | tab | newline | comment ;

program     = { definition | statement } ;

definition  = variable_def | constant_def | array_def | proc_def ;
variable_def = "VARIABLE", identifier ;
constant_def = "CONSTANT", identifier ;
array_def    = "CREATE", identifier, number ;
proc_def     = ":", identifier, { definition | statement }, ";" ;

statement   = simple_stmt
            | if_stmt
            | begin_until_stmt
            | do_loop_stmt
            | tick_stmt
            | execute_stmt
            ;
statement_or_i = statement | "I"

simple_stmt = builtin
            | number
            | string
            | identifier
            ;

builtin     = "DUP" | "SWAP" | "DROP" | "ROT" | "OVER"
            | ">R" | "R>"
            | "@" | "!"
            | "+" | "-" | "*" | "/" | "%"
            | "&" | "|" | "^"
            | "=" | "!=" | ">=" | "<=" | "<" | ">"
            | "U<" | "U>"
            | "not"
            | "emit" | "." | "key"
            ;

if_stmt         = "if", statement, [ "else", statement ], "then" ;
begin_until_stmt = "begin", statement, "until" ;
do_loop_stmt    = "do", { statement_or_i }, "loop" ;
tick_stmt       = "'", identifier ;
execute_stmt    = "execute" ; 
```

## 2. Семантика

2.1 Стратегия вычислений
Постфиксная (стековая). Программа оперирует двумя стеками:

Стек данных – хранит 32-битные знаковые целые числа. Большинство слов берут аргументы с вершины стека и помещают результат обратно.

Стек возвратов – служит для хранения адресов возврата при вызовах процедур, а также для временного хранения значений (слова >R и R>).

Вычисление происходит последовательно: каждое слово (лексема) выполняется, изменяя состояние стеков и памяти.

2.2 Области видимости
Все определения (VARIABLE, CONSTANT, CREATE, : … ;) глобальны. Имя можно использовать только после его определения (программа просматривается сверху вниз).

Имена должны быть уникальными в пределах всей программы.

2.3 Типизация
Основной тип – 32-битное знаковое целое число. Литералы чисел записываются в десятичной системе.

Логические значения кодируются целыми числами: 0 – ложь, любое ненулевое значение – истина.

Строки – компилируются в последовательность байт, завершающуюся нулём. На стек помещается адрес первого байта строки.

Массивы – последовательность 32-битных ячеек памяти; На стек помещается адрес начала и длинна массива.

2.4 Литералы
Числовой литерал – последовательность цифр. Компилятор помещает его значение на стек данных.

Строковый литерал – заключается в двойные кавычки. Компилятор выделяет память для строки (байты текста + нулевой байт) и помещает на стек адрес начала строки.

2.5 Процедуры

Определение : имя ... ;

При вызове процедуры по её имени текущее значение программного счётчика сохраняется на стеке возвратов, управление передаётся на тело процедуры. ; возвращает управление.

Допускается рекурсия (прямая и косвенная).

2.6 Управляющие конструкции
Конструкция	Семантика

if statements else statements then	

Снимает со стека значение. Если значение ≠ 0, выполняет ветку после if, иначе, если есть, else.

begin statements until	

Выполняет statements, затем снимает со стека значение. Если значение = 0, повторяет цикл; если ≠ 0 – выходит.

do statements loop	

Ожидает на стеке два числа: limit и start. Счётчик цикла (недоступен напрямую) инициализируется значением start. На каждой итерации выполняется statements, затем счётчик увеличивается на число с верхушки стека данных. Цикл завершается, когда счётчик становится больше или равным limit. Внутри тела statements слово I помещает на стек текущее значение счётчика.

2.7 Слова для работы со стеком и памятью


DUP	Скопировать вершину стека (a → a a).

DROP	Удалить вершину стека (a → ).

SWAP	Поменять местами два верхних элемента (a b → b a).

OVER	Скопировать второй элемент на вершину (a b → a b a).

ROT	Вращение трёх верхних элементов (a b c → b c a).

\>R	Переместить значение со стека данных на стек возвратов.

R>	Переместить значение со стека возвратов на стек данных.

@	Прочитать 32-битное слово из памяти по адресу, взятому со стека; поместить значение.

!	Записать 32-битное слово в память: адрес (со стека) и значение (со стека).

emit	Вывести символ по коду (берётся 8 младших бит вершины стека).

.	Вывести целое число (со знаком) из вершины стека.

key	Прочитать один символ из стандартного ввода; дополнить нулями и положить на стек.

2.8 Арифметические и логические операции
Все бинарные операции снимают два операнда. Результат помещается на стек.


\+	сложение

\-	вычитание (первый операнд минус второй)

\*	умножение

/	целочисленное деление 

%	остаток от деления (знак остатка совпадает со знаком делимого)

&	побитовое И

|	побитовое ИЛИ

^	побитовое исключающее ИЛИ

not	побитовое НЕ

Сравнения (результат: 0 – ложь, 1 – истина):

=	равно

!=	не равно

<	меньше (знаковое)

\>	больше (знаковое)

<=	меньше или равно

\>=	больше или равно

U<	меньше (беззнаковое)

U>	больше (беззнаковое)

2.9 Косвенный вызов

' имя – помещает на стек адрес процедуры имя.

execute – снимает со стека адрес и выполняет процедуру по этому адресу.

# Организация памяти

## 1. Модель памяти процессора

Процессор реализован как 32-битная RISC‑V подобная машина с **единым адресным пространством памяти** (архитектура Фон‑Неймана).

### 1.1 Доступные регистры

Программисту (в рамках Forth-программ) доступны следующие регистры (они же используются компилятором неявно):

| Регистр | Назначение |
|---------|------------|
| `x0`    | Hw нулевой регистр |
| `x1`    | Основной временный регистр |
| `x2`    | Временный регистр |
| `x3`    | Временный регистр |
| `x4`    | Временный регистр |
| `x5`    | Временный регистр |
| `x26`   | Нижняя граница буфера ввода|
| `x27`   | Верхняя граница буфера ввода|
| `x28`   | Указатель чтения из буфера ввода |
| `x29`   | Указатель записи в буфер ввода |
| `x30`   | Указатель стека данных |
| `x31`   | Указатель стека возвратов |

Регистры x6–x25 не используются.

**Стек данных** расположен в памяти, адресуемой через x30 (указатель стека). Стек растёт вниз (к младшим адресам).  
**Стек возвратов** (для адресов вызовов и временного хранения) использует регистр x31. Стек растёт вниз (к младшим адресам).

### 1.2 Память
Память представляет собой **одно непрерывное адресное пространство**, в котором последовательно размещаются:

- **Область данных** (статические данные: переменные, константы, массивы, строки, буферы ввода, векторы прерываний).
- **Область кода** (машинный код процедур и основной программы).
 

### 1.3 Адресация

- **Регистровая прямая**: большинство RISC инструкций работают с регистрами.

- **Непосредственная (immediate)**: для констант и литералов, встраиваемых в инструкции (например, addi).

- **Относительная (PC-relative)**: для переходов (jal, jalr, branch).

- **Базовая + смещение**: для доступа к памяти (lw, sw, lb, sb). Базовый адрес берётся из регистра, смещение – из immediate поля.

## 2. Отображение программы и данных (на уровне компиляции и загрузки)

### 2.1 Общая схема размещения

При компиляции Forth-программы формируется **единый образ памяти**, состоящий из трёх последовательных областей (все в одном адресном пространстве):

| Адрес (байты) | Содержимое |
|---------------|------------|
| `0x0000` | вектор прерывания | 
| `0x0004-0x0023` | обработчик прерывания
| `0x0024-0x00A3` | Программно-управляемый кольцевой буфер ввода (32 слова) |
| `0x00A4-proc_addr - 1` | Переменные, константы, массивы, строки
| `proc_addr - ...` | Машинный код процедур и основной программы | Выровнено |

## 3. Работа с литералами, константами, переменными на этапах компиляции и исполнения

### 3.1 Литералы (числовые и строковые)

**Числовой литерал** (например, 123):

- Не сохраняется в статическую память. Вместо этого генерируется последовательность инструкций, которая загружает значение непосредственно в регистр. Затем полученное значение помещается на стек данных.

**Строковый литерал** ("hello"):

- Строка преобразуется в последовательность байт, дополняется нулём, выравнивается до 4 байт и помещается в область данных. Адрес начала строки запоминается.
В месте использования литерала генерируется код загружающий **адрес** , который помещается на стек.

### 3.2 Константы (CONSTANT)

- Во время компиляции CONSTANT name только резервирует ячейку в статической памяти (4 байта), но не записывает туда значение.
Во время исполнения первая встреча CONSTANT name загружает текущее значение со стека данных и сохраняет его по адресу константы .
- **Использование имени константы** в выражении: генерируется код, который загружает **значение** из статической памяти по адресу константы и помещает его на стек.

### 3.3 Переменные (VARIABLE)

- При определении VARIABLE x резервируется ячейка в области данных (4 байта).
- Использование имени переменной в коде: генерируется код, который помещает на стек **адрес** переменной (как литерал). Для взаимодействия используются ! и @

Отображение на регистры не производится – все переменные живут в статической памяти. Регистры используются только для временного хранения.

### 3.4 Массивы (CREATE)

- Во время компиляции при встрече с CREATE name size выделяется size слов в статической памяти. Адрес начала сохраняется.
- Во время исполнения CREATE name size кладет на стек данных сначала адрес массива а затем его длину
- Дальнейшее обращение по имени кладет на стек данных адрес начала массива

### 3.5 Инструкции

- Инструкции генерируются компилятором из Forth‑слов и размещаются последовательно в памяти кода.
- Каждая инструкция кодируется 32 битами (RISC). Порядок полей – младший бит первый (little-endian).

### 3.6 Процедуры

- При компиляции процедуры (: name ... ;) её тело преобразуется в машинный код, который записывается в области инструкций. Её адрес сохраняется.
- При вызове процедуры по имени в коде компилятор вставляет инструкцию jal с вычисленным смещением до адреса процедуры. Обработка jal происходит после компиляции всего остального.
- Во время исполнения инструкция jal сохраняет адрес возврата в регистре x1 и передаёт управление. Начало процедуры сохраняет x31 на стек возвратов, конец восстанавливает и возвращается через jalr.
- Также вызов процедуры возможен через \' name и execute. \' name кладет на стек данных адрес начала процедуры. execute снимает это значение со стека и вызывает процедуру через jalr

### 3.7 Прерывания

- Прерывания обрабатываются на уровне CPU: при поступлении сигнала от внешнего устройства и флаг, запрещающий прерывания, нулевой CPU переходит в специальный режим.
- В этом режиме сначала проверяется все ли команды были исполнены и сохранены. Если нет, но выставляется специальный флаг, запрещающий подгрузку новых инструкций 
- После за 4 так происходит следующее. Текущее значение pc cохраняется на вершину стек адресов, устройство IO выдает адрес вектора прерывания, вектор прерывания загружается из памяти в pc. Первый флаг обнуляется и выставляется второй флаг, запрещающий прерывания.
- Обработчики прерываний – обычные Forth‑процедуры, скомпилированные в код. По окончании они должны выполнить специальную инструкцию jalri для возврата из прерывания.

### 3.8 Ситуация нехватки регистров (невозможна)

- Регистров для отображения переменных не используется – все переменные размещаются в статической памяти. Поэтому проблемы нехватки регистров для переменных **не возникает**.
- Для временных значений используется стек данных, размер которого ограничен только доступной памятью (по умолчанию стек имеет большой запас, выделенный в области данных).
- Для стека возвратов и стека данных используются два выделенных регистра (x30 и x31) – их переполнение может произойти только при глубокой рекурсии или большом количестве временных значений. Компилятор не контролирует глубину стеков; это остаётся на ответственность программиста.

# Система команд


Процессор представляет собой 32‑битную RISC‑подобную машину с суперскалярной организацией. Основные характеристики:

**Система команд** – RISC‑подобная, с фиксированной длиной инструкции (32 бита). Преобладают регистр-регистровые операции, доступ к памяти только через специализированные команды загрузки/сохранения.

**Время выполнения:** все инструкции (включая умножение, деление, загрузки, сохранения, переходы, ввод‑вывод) занимают **ровно 4 такта** от момента выборки до фиксации результата. Это реализуется за счет определенных аппаратных решений, организации суперскалярности.

### Типы данных и машинные слова

- **Машинное слово** – 32 бита (4 байта). Все регистры и шины данных 32‑битные.
- **Байт** – 8 бит. Поддерживаются байтовые загрузки (lb) и сохранения (sb) с распространением знака.
- **Целочисленные типы** – 32‑битные знаковые (дополнительный код). Логические значения кодируются целыми: 0 – ложь, ненулевое – истина.
- **Адреса** – 32‑битные.

### Устройство памяти и регистров

**Память** – единое адресное пространство размером 2³² байт (адреса 32‑битные). Доступ к памяти осуществляется только через инструкции lb, lw, sb, sw. Отсутствует прямое кодирование абсолютных адресов – все обращения используют базовый регистр и смещение.

**Регистры:**
- 32 общих регистра: x0 – x31. x0 аппаратно занулён.
- Специализированные регистры (с точки зрения программной модели):
  - x30 – указатель стека данных (растёт вниз).
  - x31 – указатель стека возвратов (для адресов вызовов и временного хранения).
  - x1 – x5, x26 – x29 используются как временные или для служебных целей.
- Регистр pc (program counter) не является частью регистрового файла, но управляется инструкциями перехода.

### Адресация

Поддерживаются следующие режимы адресации:

- **Регистровая прямая** – операнды находятся в регистрах (R‑тип).
- **Непосредственная (immediate)** – константа встроена в инструкцию.
- **Относительная (PC-relative)** – для переходов (jal, ветвления), смещение относительно текущего pc.
- **Базовая + смещение** – для загрузок/сохранений: адрес = (регистр rs1) + sign‑extended immediate.


### 1.4 Устройство ввода-вывода

Ввод-вывод организован через **порты** (port), адресуемые по номерам. Реализовано два типа операций:

- **input rd** – читает 32‑битное значение с порта, номер которого находится в регистре rs1.
- **print rs1** (или print_word rd) – выводит символ (байт) или число (слово) на порт, указанный в rd.

Все операции ввода-вывода синхронизированы с тактами процессора: устройство ввода возвращает значение с максимальным тактом, не превышающим текущий.

### 1.5 Поток управления и прерывания

- **Поток управления** линейный, изменяется инструкциями переходов (beq, bne, bgt, …), безусловными переходами (jal, jalr) и специальной jalri (используется для прерываний).
- **Прерывания** вызываются внешними устройствами ввода. Процессор имеет флаг _in_trap и сохраняет контекст (pc) перед переходом к обработчику. Вектор прерывания – это адрес в памяти данных. Поддерживается только одно прерывание за раз (без вложенности).
- **Завершение программы** – инструкция halt, останавливающая выполнение.

## 2. Набор инструкций

Инструкции делятся на несколько групп: арифметико-логические (R‑тип), операции с непосредственным операндом (I‑тип), загрузки/сохранения, переходы, специальные (ввод/вывод, прерывания, останов).

**Все инструкции выполняются за 4 такта.** В течение этих четырёх тактов инструкция проходит стадии (выборка, декодирование, исполнение, запись результата).

### 2.1 Список инструкций с мнемониками и временем выполнения

| Мнемоника           | Тип      | Такты | Описание |
|---------------------|----------|-------|-----------|
| `add rd, rs1, rs2`  | R        | 4     | rd = rs1 + rs2 |
| `sub rd, rs1, rs2`  | R        | 4     | rd = rs1 - rs2 |
| `mul rd, rs1, rs2`  | R        | 4     | rd = rs1 * rs2 (младшие 32 бита) |
| `mulh rd, rs1, rs2` | R        | 4     | rd = (rs1 * rs2) >> 32 |
| `div rd, rs1, rs2`  | R        | 4     | rd = rs1 / rs2 (целочисленное) |

# Транслятор

Транслятор языка Forth  реализован в виде консольного приложения `forth_compiler.py`. Он выполняет компиляцию исходного файла в бинарный образ памяти, который может быть загружен и выполнен на целевой виртуальной машине .

## 1. Интерфейс командной строки

Запуск транслятора осуществляется из командной строки. Поддерживаются следующие параметры:

```bash
python forth_compiler.py <input_file> [-o <output_bin>] [-l <listing_file>] [--format binary|text]
```
input_file	- Обязательный. Путь к текстовому файлу с исходным кодом на языке Forth.

-o, --output-bin	Необязательный. Имя выходного бинарного файла (машинный код). По умолчанию формируется как <имя_входного_файла>.bin.

-l, --listing - Необязательный. Имя файла для сохранения листинга (текстового представления машинного кода с мнемониками). По умолчанию <имя_входного_файла>.lst.

--format	Необязательный. Формат выходного файла. Возможные значения: binary (бинарный, по умолчанию), text (шестнадцатеричные строки). Для обычного использования рекомендуется binary.


#### Выходные данные

Бинарный файл (.bin) – машинный код, упакованный как 32-битные слова в порядке little‑endian. Может быть непосредственно загружен в память целевого процессора.

Файл листинга (.lst) – текстовый отчёт, содержащий для каждой инструкции:

адрес начала программы

При возникновении синтаксической ошибки транслятор выводит сообщение в stderr и завершается с кодом ошибки.

## 2. Принципы работы транслятора
Трансляция проходит в четыре последовательных прохода по исходному коду, разделённых на этапы лексического анализа, построения таблиц символов, генерации кода и связывания.

### 2.1 Этапы трансляции
#### 2.1.1 Лексический анализ (токенизация)
Функция tokenize(source) разбивает входную строку на токены, игнорируя:

пробельные символы,

комментарии \ … (до конца строки),

вложенные комментарии ( … ),

На выходе формируется список токенов, которые подаются на вход последующим проходам.

#### 2.1.2 Первый проход (сбор информации о данных)
Цель: определить адреса всех статических данных (переменные, константы, массивы, строковые литералы) и вычислить начальный адрес кода (proc_addr).

Обрабатываются определения VARIABLE, CONSTANT, CREATE.

Для каждого имени запоминается его тип и адрес в области данных.

Строковые литералы преобразуются в последовательности байт, выравниваются до 4 байт и размещаются в области данных.

После завершения прохода proc_addr устанавливается на адрес, следующий за последним байтом данных.

#### 2.1.3 Второй проход (сбор определений процедур)
Обрабатываются определения процедур : имя ... ;. Для каждой процедуры:

Запоминается её адрес в области кода (текущее значение proc_addr).

Тело процедуры транслируется в машинный код (результат методов compile_*), который накапливается в proc_code.

Код процедуры предваряется прологом (сохранение стека возвратов) и завершается эпилогом (восстановление и возврат).

После обработки процедуры proc_addr увеличивается на длину сгенерированного кода.

На этом этапе код генерируется только для процедур.

#### 2.1.4 Третий проход.
Транслируются все оставшиеся операторы (не входящие в определения). Для каждого токена (число, встроенное слово, имя переменной, управляющая конструкция) вызываются соответствующие методы компиляции. Результирующий код добавляется в proc_code после всех процедур.

В конце прохода добавляется инструкция halt, останавливающая процессор.

#### 2.1.5 Четвёртый проход (связывание)
В сгенерированном коде остаются строковые метки – имена процедур, которые ещё не заменены на инструкции вызова. Четвёртый проход проходит по всем элементам all_code (объединённые data_code + proc_code + код верхнего уровня) и для каждой строки имя вставляет вместо неё инструкцию jal с вычисленным смещением до соответствующей процедуры.

# Виртуальная машина (CPU)
## 1. Интерфейс командной строки

Запуск виртуальной машины осуществляется командой:

```bash
python vm.py <bin_file> <pc> [-i <input_file>] [--log-level {1,2,3}]
```
bin_file - Обязательный. Путь к бинарному файлу с машинным кодом .

pc - Обязательный. Начальное значение программного счётчика в десятичном или шестнадцатеричном формате (например, 16448 или 0x4040).

-i, --input	Необязательный. Путь к текстовому файлу с входными данными для имитации ввода (формат описан ниже). Если не указан, ввод считается пустым.

--log-level	Необязательный. Уровень детализации журнала: 0 – журнал отключён (по умолчанию), 1 – только ошибки и завершение, 2 – важные события (такты, коммиты, прерывания), 3 – полная трассировка (состояния регистров, все инструкции, ввод/вывод).


#### Входные данные

Бинарный файл с машинным кодом
Файл, сгенерированный транслятором, содержит 32‑битные little‑endian слова, упакованные подряд. Первое слово находится по адресу 0x0000. Процессор загружает этот файл в свою память и начинает выполнение с адреса, заданного параметром pc.

Файл входных данных (опционально)
Файл имитирует поток ввода данных от внешних устройств (например, клавиатуры). Формат файла:

text
<тактовое_значение> <символ>;
тактовое_значение – целое число, такт процессора, на котором символ становится доступным.

символ – одиночный символ (латиница, цифра или escape‑последовательность \n, \t, \0).

Разделитель – точка с запятой ; (может быть несколько записей на одной строке).

Пример:

text
100 a; 200 b; 300 \n; 400 \0;
Внутри виртуальной машины эти данные помещаются в очередь устройства ввода, и при выполнении инструкции key или input CPU забирает символ с наиболее подходящим тактом (последний, у которого clock <= tick).

#### Выходные данные

Вывод процессора
Все данные, выводимые инструкциями emit, . , направляются в стандартный вывод (stdout). Кроме того, они дублируются в журнал.

Формат вывода:

Для emit – печатается символ (без перевода строки).

Для . – печатается десятичное целое число, за которым следует пробел.

### Симуляция

Происходит потактово. Всего 5 ( + остановка) состояний между которыми происходит переключение. 

FETCH - загрузка инструкций по значению PC в IR 1 - 4.

### 2.2 EXECUTE
В записи ROB выставляется флаг ready
А так же обновляются необходимые значения
### 2.3 COMMIT
Инструкции коммитятся по порядку их записи в буффер, так обеспечивается разрешение WAW зависимостей. Если был произведен коммит управляющей инструкции то ROB сбрасывает все записи. Также после коммита происходит BROADCAST - записи из RS ожидающие результат текущей инструкции получают его, а так же очистка RENAME для регистров и памяти если текущая инструкция последняя занявшая регистр или память.
### 2.3 TRAP HANDLE
Переход в прерывания невозможен пока все инструкции из ROB не будут выполнены.

## 3 RS
### 3.1 ISSUE
В процессе в первом свободном месте формируется запись состоящая из:

Валидности

Тега для ROB

Операнда 1

Тега Операнда 1

Готовности Операнда 1

Операнда 2

Тега Операнда 2

Готовности Операнда 2

Тега памяти

IMM значения

OPCODE FUNCT 3 и 7

Нескольких флагов

### 2.2 EXECUTE 
Выбирается 3 готовые для выполнения инструкции, только одна из которых работает с памятью. Каждой - назначается АЛУ. После выполнения происходит запись в ROB пот тегу и еще один BROADCAST. 
### 2.3 COMMIT
Могут появится новые готовые к выполнению инструкции засчет BROADCAST  в COMMIT
### 2.4 TRAP HANDLE
Переход в прерывания невозможен пока все инструкции из RS не будут выполнены.

## RENAME (REG и MEM)
Таблицы переименований, хранящие последний ROB-тег инструкции, пишущей в этот регистр / в память по этому регистру, а так же готовность регистра. Так разершаются RAW зависимости.
